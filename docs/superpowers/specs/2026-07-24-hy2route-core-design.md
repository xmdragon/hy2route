# hy2route-core 精简透明代理设计

日期：2026-07-24
状态：已确认

## 1. 背景

当前 hy2route 由 dnsmasq、chinadns-ng、Xray、nftables、procd 和若干生成及监督脚本组成。它能完成透明代理，但存在以下问题：

- 运行组件和部署依赖较多，任一文件遗漏都可能破坏启动链。
- Xray 包含大量本项目不需要的协议和路由功能。
- 单纯按 DNS 返回的 IPv4 分流，会被 DNS 负载均衡和共享 CDN 地址误导。
- 多进程之间存在重复缓存、配置同步和额外内存开销。

本设计用一个精简的 Go 常驻进程替换 Xray 与 chinadns-ng，同时保留 OpenWrt 自带 dnsmasq、nftables、procd 和 UCI。

## 2. 目标

第一版必须满足：

- 仅支持 IPv4。
- 国内域名和 China IPv4 直连。
- 海外域名和非 China IPv4 默认通过 HY2。
- TCP 支持 HY2 直接出口，或 `HY2 -> HTTP CONNECT/SOCKS5 落地 -> 目标`。
- UDP 国内直连、海外通过 HY2 服务器直接出口，永不经过落地代理。
- 保留精简 UCI 配置、CLI 和 LuCI 页面。
- HY2 故障时采用可用性优先策略，海外流量降级为直连。
- 减少常驻进程、外部包依赖和内存占用。
- 国内直连路径尽量留在内核，避免用户态代理影响千兆吞吐。

## 3. 非目标

第一版不支持：

- IPv6 或 AAAA 转发。
- VLESS、VMess、Trojan 等其他代理协议。
- SOCKS5 UDP 落地、HTTP/3 代理、反向代理或服务端模式。
- 多级任意代理链、脚本规则、正则域名规则。
- DNSSEC 验证、DoH 服务端、广告过滤。
- 流量统计数据库或常驻 Web 后端。
- 自研 QUIC、TLS 或密码学实现。

## 4. 总体架构

OpenWrt 运行时组件：

1. dnsmasq：DHCP、局域网主机名和 DNS 缓存。
2. `hy2route-core`：智能 DNS、透明 TCP/UDP、HY2 客户端及落地握手。
3. nftables：透明转发、内核直连旁路和动态 IPv4 集。
4. procd：进程生命周期、崩溃拉起和资源限制。

项目只新增一个常驻进程。`hy2route-core` 使用成熟的 Go QUIC、TLS 和 Hysteria2 协议库并静态编译，路由器不需要单独安装这些库。

```text
DNS:
客户端 -> dnsmasq -> hy2route-core:1053

国内 TCP:
客户端 -> nftables 内核旁路 -> 目标

海外 TCP（HY2 直出）:
客户端 -> nftables TPROXY -> hy2route-core -> HY2 -> 目标

海外 TCP（落地）:
客户端 -> nftables TPROXY -> hy2route-core -> HY2
       -> HTTP CONNECT/SOCKS5 落地 -> 目标

海外 UDP:
客户端 -> nftables TPROXY -> hy2route-core -> HY2 UDP -> 目标
```

## 5. 智能 DNS

dnsmasq 将上游 A 和 AAAA 查询交给 `hy2route-core`。核心：

- 正常处理 A 查询。
- 对 AAAA 返回 NODATA，不产生 IPv6 地址。
- 支持本地 UDP 和 TCP DNS 请求。
- 国内上游默认使用运营商或用户指定的国内 DNS。
- 可信海外 DNS 查询通过 HY2 发送，避免本地链路污染。

域名分为三类：

### 5.1 中国域名

匹配中国域名后缀树的请求只查询国内 DNS。返回的 IPv4 按 DNS TTL 加入动态直连候选。即使地址不在 China IPv4 库中，该域名的结果仍按域名规则直连；若该地址同时被代理域名使用，则改为进入核心检查，而不是直接内核旁路。

### 5.2 明确海外域名

匹配海外规则的请求通过 HY2 查询可信 DNS，返回的 IPv4 按 TTL 加入动态代理集合。

### 5.3 未分类域名

国内 DNS 与可信海外 DNS 并发查询：

- 国内答案中的所有可用 A 记录都属于 China IPv4 库时，采用国内答案并标记直连。
- 国内答案包含非 China IPv4、无有效 A、超时或错误时，采用可信答案并标记代理。
- 两侧都失败时返回 DNS 失败，不缓存伪造的成功结果。

核心维护有容量上限的 DNS 缓存和域名/IP映射。缓存过期时重新计算该 IPv4 的有效关系，并同步更新动态 nftables 元素，防止历史 CDN 地址永久影响分流。

HY2 故障进入降级状态后，海外 DNS 先直接查询同一可信 DNS；若仍不可达，再查询国内 DNS。所有降级和恢复事件写入限速日志。该策略优先保证可用性，可能产生 DNS 泄漏或受到本地 DNS 污染。

## 6. 分流判定

判定优先级从高到低：

1. 用户显式域名或 IPv4/CIDR 规则。
2. 当前 TCP 连接的 TLS SNI 或 HTTP Host。
3. DNS 学到且仍在 TTL 内的域名/IP关系。
4. China IPv4 前缀库。
5. 默认海外代理。

核心只读取 TCP 开头受限长度的数据以提取 TLS SNI 或 HTTP Host，不解密内容、不做深度协议识别。共享 CDN 地址同时属于直连和代理映射时，当前连接的 SNI/Host 决定结果；没有域名信息时，显式规则优先，否则冲突地址采用代理，以避免意外绕过代理。

nftables 使用互斥的快速路径状态：

- 仅存在直连关系的 IPv4 进入 `direct4`，可以在内核中旁路。
- 存在代理关系的 IPv4 进入 `inspect4`，其匹配优先于 `china4` 和 `direct4`。
- 同时存在直连与代理关系的共享 IPv4 只进入 `inspect4`，由核心结合 SNI/Host 判断。
- 用户显式 IPv4/CIDR 规则位于动态集合之前，保持最高优先级。

因此，某个 China IPv4 若同时被明确海外域名使用，不会在嗅探前被 `china4` 提前放行。UDP 没有 SNI/Host 可用，遇到共享或冲突 IPv4 时采用代理。

客户端绕过路由器 DNS 使用 DoH 时，TLS SNI 仍可提供域名判定。若客户端同时使用 ECH，核心只能按 IPv4 判断，不保证域名级准确性。

## 7. 数据结构与更新

项目维护两类独立源数据：

- 中国域名后缀树源列表；
- China IPv4 前缀树源列表。

构建时将两类源数据合成为一个带 SHA-256 校验的紧凑 `routing.bin`，运行时只读取这一份快照，避免双文件版本错配和额外内存占用。数据仍可独立更新，无需替换核心程序；更新流程先验证源数据和新快照，再原子替换 `routing.bin`，失败继续使用上一版本。

`hy2route-core` 通过 netlink 维护 nftables 动态集合及超时，不为每次 DNS 响应启动 `nft` 子进程。

## 8. TCP 与落地代理

核心保留长期 HY2/QUIC 会话，并为每个透明 TCP 连接打开独立 QUIC stream：

- HY2 直出模式：stream 的目标就是最终 IPv4 和端口。
- HTTP 落地模式：HY2 stream 先连接落地地址，然后执行最小 HTTP CONNECT 握手；只支持必要的 Basic 认证。
- SOCKS5 落地模式：HY2 stream 先连接落地地址，然后执行 SOCKS5 CONNECT；支持无认证和用户名/密码认证。

握手失败不会在同一连接内改用另一种落地协议。核心记录结构化错误，并依据故障状态决定后续新连接是否降级直连。

## 9. UDP

核心按源地址、源端口、目标地址、目标端口维护有界 UDP 会话：

- 没有代理或冲突关系的 China IPv4 由 nftables 直接放行。
- 其他 IPv4 使用 HY2 UDP 转发，由 HY2 服务器直接访问目标。
- HTTP/SOCKS5 落地配置不影响 UDP。
- 会话达到容量上限时淘汰最久未使用项。
- 空闲会话按可配置超时释放，配置范围设置安全上下限。

## 10. 故障与恢复

故障域相互隔离：

- HY2 或落地故障不影响国内内核直连。
- HY2 故障时，海外 TCP、UDP 和可信 DNS转为直接出口。
- 国内 DNS 故障时，不把未经 China IPv4 验证的未知域名答案标记为直连。
- nftables 代理链由核心定期续期的短 TTL 心跳启用；核心异常退出且心跳过期后，透明流量自动回到内核直连。
- dnsmasq 将国内 DNS 作为核心无响应时的后备上游，不得形成 DNS 循环；procd 自动重启核心并记录阶段化错误。
- 日志对重复错误限速，避免故障期间写满存储或日志缓冲。

降级采用带迟滞的状态机：

- 连续多个健康探测失败后进入直连降级，避免单次抖动切换。
- 冷却时间结束且连续多个探测成功后恢复 HY2。
- 现有连接不强制迁移；状态变化只影响新连接和新 DNS 查询。

## 11. 配置与管理

UCI 和 LuCI 仅暴露：

- HY2 地址、端口、认证、SNI 和 TLS 校验。
- `direct`、`http` 或 `socks5` TCP 出口模式。
- HTTP/SOCKS5 落地地址、端口和认证。
- 国内 DNS 与可信海外 DNS。
- 自定义直连/代理域名、IPv4 和 CIDR。
- 故障时海外直连开关；第一版默认开启。

缓存容量、UDP 空闲时间、健康检查和资源上限提供经过验证的默认值，放在折叠的高级区域。认证信息保存在 root 可读的 UCI 配置中，不放在进程命令行或普通日志中。

CLI 保留：

- `check`：离线校验配置、数据文件、端口和内核能力。
- `status`：进程、HY2、DNS、降级状态和资源摘要。
- `test`：分别测试国内直连、HY2 直出、落地、UDP 和 DNS。
- `reload`：校验成功后原子加载新配置。

## 12. 性能与资源控制

国内路径不经过 Go 用户态：

- 没有代理冲突的 China IPv4、DNS 确认的国内地址和显式直连规则由 nftables 直接旁路。
- `inspect4` 在 `china4` 和动态 `direct4` 之前匹配，避免共享 CDN 被提前直连。
- 直连路径允许使用 OpenWrt flow offload。
- 仅海外和未知目标进入 TPROXY。

核心使用：

- 长期 HY2/QUIC 会话和独立 stream。
- 有界 DNS、域名映射、UDP 会话和连接状态。
- 复用的分级缓冲池。
- 批量 netlink 更新。
- 低频、限速的健康探测和日志。

目标负载为 20 台设备、国内最高 1000 Mbps、海外最高 300 Mbps。真机验收目标：

- 国内直连吞吐相对关闭 hy2route 的基准下降不超过 10%。
- 在服务器和线路基线达到 300 Mbps 时，HY2 吞吐达到 280 Mbps 以上。
- 常规家庭负载下核心 RSS 不超过 40 MB。
- 空闲 RSS 目标不超过 24 MB。
- 压力测试峰值 RSS 不超过 64 MB，负载结束后能够回落。
- 连续运行 72 小时，内存和连接表无持续增长。

这些是验收门槛，不是脱离硬件和线路条件的预先保证。若未达标，必须用 CPU、内存和网络 profile 定位瓶颈后再优化。

## 14. 迁移与回滚

迁移分阶段执行：

1. 构建 arm64 单一静态二进制并完成离线测试。
2. 在路由器备用端口运行，与现有 Xray 服务并行。
3. 验证国内/海外 DNS、HY2 直出、两类落地、UDP 和故障放行。
4. 只将一台测试客户端切换到新 nftables 标记和 DNS。
5. 观察稳定后切换全网。
6. 保留旧 Xray、chinadns-ng、配置和启动脚本，任何验收失败都原子回滚。
7. 新核心通过性能与 72 小时稳定性验收后，才卸载 Xray 和 chinadns-ng。

每次正式切换前都创建 root 专用的时间戳备份，记录原文件是否存在。回滚必须同时恢复文件、UCI、dnsmasq 上游、nftables 和 procd 服务状态。

## 15. 完成标准

只有同时满足以下条件，才能认为替换完成：

- 运行时不再依赖 Xray 和 chinadns-ng。
- 所有目标流量路径和故障降级测试通过。
- DNS 负载均衡及共享 CDN 测试没有已知误分流。
- 真机吞吐、内存和 72 小时稳定性达到第 12 节门槛。
- UCI、LuCI、CLI、升级和回滚文档完整。
- 旧服务在稳定观察期结束前保持可恢复。
